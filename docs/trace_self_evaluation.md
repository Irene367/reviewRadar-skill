# ReviewRadar TRACE 自评

## Trust 信任

ReviewRadar 要求先建立证据编号，再进行结论输出。商品评论使用 `R01、R02...`，客服记录使用 `C01、C02...`，售后反馈使用 `A01、A02...`。报告中的问题聚类、高风险预警和今日三项行动都应尽量引用编号证据，帮助商家追溯结论从哪里来。

风险判断不只写“高 / 中 / 低”，而是引入 5 个评分维度：出现频次、情绪强度、转化影响、售后成本、扩散风险。综合风险分用于辅助排序，避免完全主观判断。若证据不足，必须标注“证据不足”“不确定”或“低置信度”。

Top 3 高风险问题增加“风险评分分项依据”，把总分拆成出现频次、情绪强度、转化影响、售后成本、扩散风险五项，并说明评分来自哪些评论或客服证据。这增强 Trust，让风险判断更可解释，也避免用户误以为风险分是精确统计。

在隐私处理上，ReviewRadar 不复述姓名、手机号、地址、订单号、账号 ID、物流单号等敏感信息。即使用户粘贴了这类内容，分析时也应忽略或替换为 `[已脱敏]`，只保留与口碑问题相关的语义。

## Reliability 可靠性

ReviewRadar 使用固定报告结构：总览、问题聚类、高风险预警、差评回复、客服知识库更新包、详情页优化、售后 SOP、口碑日报、今日三项优先行动。这个结构能让一人卖家每天用同一套格式做复盘，也方便后续横向对比。

简版 / 完整版输出模式增强 Reliability 和 Applicability。简版保留口碑总览、Top 3 问题、高风险预警和今日三项行动，适合日常快速查看；完整版保留 9 个模块，适合深度诊断、周报复盘和比赛 Demo。两种模式都保留证据编号和风险等级，避免因为压缩输出而丢失关键判断依据。

面对异常输入时，Skill 有明确兜底：输入为空时要求补充数据；样本过少时输出报告但提示“样本不足，结论仅供参考”；输入混合时先按来源或问题类型粗分；字段不完整时只分析已有信息，不编造订单、物流、投诉、退货率、转化率和平台规则。

## Applicability 适用性

ReviewRadar 聚焦一人电商公司每天都会遇到的评论、客服、售后场景，而不是抽象舆情分析。它把“用户为什么不满意”拆成可处理的经营动作：客服怎么回复、知识库补什么、详情页怎么改、售后 SOP 怎么走、今天先处理哪三件事。

快速使用入口增强 Applicability。用户可以直接用“完整口碑诊断”“只看差评风险”“生成客服知识库”“优化商品详情页”“生成今日行动清单”“低样本快速判断”等触发句开始工作，不需要先理解完整 Prompt 结构。

品类适配提示增强 Applicability，使 Skill 不局限于女装 Demo。ReviewRadar 会先识别或推测商品品类，再按女装、美妆、食品、家居、数码配件、宠物用品、母婴用品、文创用品等不同维度聚类问题；如果品类无法判断，则回退到质量、描述一致性、履约、包装、客服、售后、价格预期等通用维度。

“今日三项优先行动”要求每个动作都能由店主本人独立完成，避免输出“重构供应链”“全面升级客服体系”这类大而空的建议。它更适合 OPC 的真实工作节奏。

## Compliance 规范性

ReviewRadar MVP 不接平台 API，不做非授权爬虫，不绕过平台登录验证，不自动采集用户隐私。数据来源限定为用户手动粘贴、用户主动导出的 CSV/Excel/文本内容和用户粘贴的客服聊天记录。

输出上，ReviewRadar 不诱导刷评，不建议删除差评，不提供控评、压评、恶意投诉、攻击用户、虚假宣传或欺骗性营销话术。遇到这类请求时，应拒绝违规部分，并改为提供合规的服务改进、公开回复、知识库更新和售后 SOP 建议。

## Effectiveness 有效性

ReviewRadar 的有效性来自“把团队能力压缩成一个人可执行的工作流”。传统小团队中，客服负责记录问题，运营负责看数据，品控负责定位原因，文案负责改详情页，主管负责制定 SOP。ReviewRadar 将这些动作整合成一份口碑运营报告，让一人公司也能按优先级执行。

报告最终不只告诉商家“大家在抱怨什么”，而是进一步给出“今天先改哪里、怎么回复、FAQ 加什么、详情页文案怎么写、售后如何处理、哪些问题本周复盘”。这使它更接近口碑运营决策 Skill，而不是普通评论总结工具。
